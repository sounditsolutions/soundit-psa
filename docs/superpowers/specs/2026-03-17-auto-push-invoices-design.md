# Auto-Push Invoices to Billing Backend on Generation

**Date:** 2026-03-17
**Bead:** psa-a33
**Status:** Approved

## Problem

Invoices generated by the billing cron (`billing:generate`) are created as Draft. Pushing them to QBO or Stripe requires either manual action (per-invoice button or bulk select) or the global auto-push cron (every 4 hours). Some profiles need invoices pushed immediately upon generation without waiting for the 4-hour window.

## Solution

A per-profile `auto_push_mode` setting that controls whether and how invoices are pushed after generation. A queued job dispatched from `InvoiceObserver::created()` handles the push. Separate notification event types alert staff when generation or push fails.

## Data Model

### Migration

**`recurring_invoice_profiles` table:**
- Add `auto_push_mode` column: `string`, nullable, default `null`

### New Enum: `AutoPushMode`

New enum at `app/Enums/AutoPushMode.php`:
- `Push` (`'push'`) ŌĆö Push to billing backend (finalize in Stripe, create in QBO)
- `PushAndSend` (`'push_and_send'`) ŌĆö Push and send invoice email via Stripe. For QBO clients, behaves the same as `Push` (QBO has no send-via-API).

A `null` value on the profile means auto-push is disabled.

### NotificationEventType Enum

**Two new values:**
- `InvoiceGenerationFailed` ŌĆö label: "Invoice generation failed"
- `InvoicePushFailed` ŌĆö label: "Invoice push failed"
- Both default to enabled via `defaultEnabled()` (billing failures are important for all staff)

## Auto-Push Flow

### Trigger

`InvoiceObserver::created()` checks whether auto-push applies:

```php
if ($invoice->profile_id && $invoice->profile?->auto_push_mode) {
    PushInvoiceToBilling::dispatch($invoice)->afterCommit();
}
```

**Null guard:** Invoices created without a profile (portal prepay purchases, Stripe imports) are skipped ŌĆö they have no `profile_id`.

**afterCommit timing:** Laravel's `afterCommit` waits for the outermost `DB::transaction()` closure to commit. Since `generateInvoice()` wraps the entire creation flow (Invoice::create ŌåÆ InvoiceLine rows ŌåÆ total updates) in a single transaction, the job won't execute until all lines and totals are persisted.

**"Generate Now" button:** Also fires `InvoiceObserver::created()`, so auto-push applies to manually triggered generation too. This is intentional ŌĆö the user enabled auto-push on the profile.

**No conflict with `updated()` observer:** The push job changes invoice status to Synced, which triggers `InvoiceObserver::updated()`. This is harmless ŌĆö the existing `updated()` only acts on Paid/Void transitions.

### PushInvoiceToBilling Job

New queued job at `app/Jobs/PushInvoiceToBilling.php`.

Properties: `$tries = 2`, `$timeout = 60`.

Steps:
1. Reload invoice with `client` and `profile` relationships
2. **Guard:** Re-check `$invoice->status === InvoiceStatus::Draft` before proceeding. If status has changed (e.g., 4-hour cron already pushed it), log and return. This prevents duplicate invoices in the billing backend.
3. Read `auto_push_mode` from the profile to determine send behavior
4. Determine billing backend and push:
   - If client has `stripe_customer_id` ŌåÆ call `StripeSyncService::pushInvoiceToStripe($invoice, sendEmail: $mode === AutoPushMode::PushAndSend)`
   - Else if client has `qbo_customer_id` ŌåÆ call `QboSyncService::pushInvoiceToQbo($invoice)` (send not applicable)
   - Else ŌåÆ log warning, dispatch `InvoicePushFailed` notification ("Client not mapped to billing backend"), return
5. On success: invoice status is `Synced` (handled by existing push methods)
6. On failure: catch exception, dispatch `InvoicePushFailed` notification with error message

### Generation Failure Notifications

In `BillingService::generateInvoicesForDueProfiles()`, the existing catch block logs errors. Add a call to `NotificationService::notifyInvoiceGenerationFailed()` in the catch block.

## Notification Methods

### NotificationService

**`notifyInvoiceGenerationFailed(RecurringInvoiceProfile $profile, string $error): void`**
- Queries users opted into `NotificationEventType::InvoiceGenerationFailed`
- Dispatches `SendTicketNotification` for each with null ticket/actor IDs and JSON context
- Context includes: profile name, contract name, client name, error

**`notifyInvoicePushFailed(Invoice $invoice, string $backend, string $error): void`**
- Queries users opted into `NotificationEventType::InvoicePushFailed`
- Dispatches `SendTicketNotification` for each with null ticket/actor IDs and JSON context
- Context includes: invoice number, client name, backend name (QBO/Stripe), error

Both follow the existing pattern used by `notifyPrepayLowBalance()` ŌĆö null ticket/actor IDs with JSON context string are already supported by `SendTicketNotification`.

## UI Changes

### Profile Detail Page (`profiles/show.blade.php`)

- Three new bulk actions in the existing bulk action dropdown:
  - "Enable auto-push (push only)"
  - "Enable auto-push (push and send)"
  - "Disable auto-push"
- Bulk action values added to `bulkAction()` validation rule

### User Notification Preferences

Two new checkboxes in the notification preferences UI (wherever `NotificationEventType` values are rendered):
- "Invoice generation failed"
- "Invoice push failed"

These are rendered automatically from the enum if the existing UI iterates `NotificationEventType::cases()`.

## Backend Determination Logic

Matches the existing priority in `InvoiceController::bulkAction()`:
- Stripe takes precedence if client has `stripe_customer_id`
- Falls back to QBO if client has `qbo_customer_id`
- If neither, push fails with a descriptive error

The `sendEmail` parameter on `pushInvoiceToStripe()` already exists. For QBO, the `PushAndSend` mode is treated identically to `Push` (QBO does not support sending invoices via API).

## Scope Boundaries

- The existing global auto-push cron (every 4 hours) is unchanged and continues to work independently
- Per-profile auto-push is additive ŌĆö the `PushInvoiceToBilling` job re-checks Draft status before pushing, so if the 4-hour cron already pushed the invoice, the job is a no-op
- No changes to the manual push buttons or bulk push action
- No changes to invoice generation logic in `BillingService::generateInvoice()`
